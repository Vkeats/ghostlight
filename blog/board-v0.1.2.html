The Board v0.1.2: Dispatch Workflow Refinements

The Board v0.1.2 is out. This release is about preparation — tightening up the dispatch workflow, cleaning up code health, and getting the system ready for onboarding next week. Consolidated views, smarter SMS handling, and a more maintainable codebase.

Getting ready for onboarding

There's a difference between software that works and software that's ready for new users. The Board has been running live, but onboarding means putting the system in front of dispatchers who haven't seen it before. Every rough edge matters more when someone is forming their first impression.

This release focuses on the things that need to be solid before that happens — views that make sense on first use, SMS workflows that don't require explanation, and code underneath that won't surprise anyone with state bugs during a demo.

What shipped

Version 0.1.2 includes 20+ targeted improvements across dispatch workflows, communications, and infrastructure. Here's what shipped.

Dispatch Views

  • Consolidated calendar view into card view with call type and date grouping — fewer clicks to see the same information
  • Position cards now always show call duration — no more guessing at shift length
  • Added group-by-dispatcher view to jobs list — helps multi-dispatcher operations track who's handling what
  • Setup and strike call types now handled correctly in hours calculator and timeline — previously treated as standard calls

SMS & Communications

  • Dispatch log now classifies SMS by type — offer, confirmation, decline, inquiry — with pipeline ordering based on intent
  • Clear button on comms action items for irrelevant messages — stops noise from blocking the needs-action queue
  • SMS conversations no longer auto-mark as processed when viewed — dispatcher control over workflow state
  • Fixed SMS toggle buttons in spreadsheet view not updating visually — UI was out of sync with state

UI Fixes

  • Needs Action section no longer collapses unexpectedly
  • Toolbar layout stabilized, duplicate stats removed
  • Fixed loading state in PositionHeatmapGrid
  • Color temperature slider added — warm mode overlay for late-night dispatch shifts
  • Streamlined nav menu for cleaner navigation

Code Health

  • Added ESLint to the project and fixed a critical React hooks violation that was causing state bugs
  • Resolved exhaustive-deps warnings across the entire codebase — 10 batched pull requests to clean up all hook dependencies
  • Composite indexes added to the database, plus an N+1 query fixed in the board jobs API — performance improvements for multi-job views

Infrastructure

  • Fixed Docker build by skipping lifecycle scripts in production deps stage
  • Fixed backup service database filename — backups were writing to the wrong path

Why SMS classification matters

One of the less obvious improvements in this release is SMS intent classification. When a dispatcher sends bulk offers to 30 workers, replies come back in a flood. Some are acceptances. Some are declines. Some are questions about the shift details. Others are completely unrelated messages that happened to arrive at the same time.

The dispatch log now classifies each incoming SMS by type and orders them according to where they sit in the dispatch pipeline. Offers that need responses surface first. Confirmations get grouped together. Irrelevant messages can be cleared with a button press instead of cluttering the action queue.

It's a small change in code, but it saves dispatchers from manually sorting through message threads when they're trying to fill a crew call before curtain.

Code health isn't optional

Adding ESLint to the project wasn't glamorous, but it was necessary. The React hooks violation it caught had been causing intermittent state bugs that were hard to reproduce. The exhaustive-deps warnings across the codebase represented dozens of places where hook dependencies weren't correctly specified — each one a potential source of stale closures or missed re-renders.

Fixing all of it took 10 pull requests and a lot of careful refactoring. But now the codebase has guardrails. New code gets linted before it ships, and the hooks rules catch problems before they ship.

Technical debt compounds. Paying it down early keeps the system maintainable.

What's next

Onboarding is next week. After that, the roadmap opens up:

  • Enhanced reporting — workforce utilization, permit hour tracking, per-position analytics
  • Multi-local architecture — right now The Board is single-tenant; scaling to support multiple locals with data isolation
  • Mobile-first improvements for the worker portal — crew members interact with the system from their phones, and the experience should reflect that
  • Expanded automation in the dispatch pipeline — more decisions the system can make on behalf of the dispatcher based on configured rules

Meanwhile, The Guild continues in preview. If you're running auditions or managing casting for a theatre company, take a look at the-guild.app.

The short version

The Board v0.1.2 ships 20+ improvements to get the system onboarding-ready. Consolidated views, smarter SMS classification, and a cleaner codebase. Next week, new dispatchers sit down with it for the first time.
